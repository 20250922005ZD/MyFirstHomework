# 基金课题项目查询工具设计文档

## 1. 项目背景
科研工作者在查询基金课题项目时，通常需要分别访问多个官方站点，例如国家自然科学基金、国家社科基金、教育部人文社科项目库等。不同站点的查询入口、字段命名、筛选方式和结果展示差异较大，导致以下问题：

- 单个来源可查，但跨来源横向对比困难。
- 同一检索意图需要重复输入、重复筛选。
- 结果字段不统一，难以快速判断项目价值。
- 查到结果后还需要逐条回源查看详情，效率较低。

本项目拟建设一个“省部级以上基金课题项目查询工具”的最小版本，先聚焦国家级公开项目数据，通过统一抓取、标准化入库和统一检索，帮助科研工作者在一个页面内完成项目查询和详情查看。

## 2. 项目目标
本项目第一版目标是在两天内完成一个最小可用版本，满足“可抓取、可查询、可查看详情”的基本能力。

第一版目标如下：

- 聚合 1 到 2 个国家级公开项目来源。
- 支持科研工作者通过关键词统一查询项目。
- 支持按来源和立项年份进行基础筛选。
- 支持查看项目详情及原始来源链接。
- 支持通过脚本手动触发一次抓取更新。
- 支持将抓取数据标准化保存，避免重复数据。
- 在无数据库环境下仍可通过本地快照展示真实抓取结果。

## 3. 非目标
以下内容明确不属于第一版范围：

- 不做用户登录、注册、权限系统。
- 不做支付、会员、配额系统。
- 不做完整管理后台。
- 不做实时全网搜索。
- 不做全部省部级以上来源一次性接入。
- 不做 Excel/CSV 导出。
- 不做收藏、订阅、提醒。
- 不做复杂全文检索和中文分词排序。
- 不处理登录态、验证码、受限访问的数据源。

## 4. 目标用户
第一版目标用户为科研工作者，具体包括：

- 高校教师
- 科研院所研究人员
- 项目申报辅助人员
- 学科建设与科研管理相关人员

这类用户的共同特点是：

- 需要快速查询某个主题、领域、负责人或单位的既有立项情况。
- 更关注查询效率、字段完整度和来源可信度。
- 更希望“一个入口完成多来源检索”，而不是逐站点重复操作。

## 5. 核心使用场景
### 5.1 按主题找已立项项目
用户输入某个研究主题词，例如“人工智能教育应用”，系统返回已抓取来源中与该主题相关的项目列表。

### 5.2 按负责人查询
用户输入某位负责人姓名，查询其在已接入来源中的项目情况。

### 5.3 按单位查询
用户输入高校或研究机构名称，查询该单位在相关项目中的立项记录。

### 5.4 按年份缩小范围
用户先输入关键词，再通过立项年份筛选，快速缩小结果范围。

### 5.5 回源核验
用户在结果列表中找到感兴趣项目，进入详情页查看完整字段，并点击原始来源链接进行核验。

## 6. 使用流程
### 6.1 用户流程
1. 用户打开系统首页。
2. 在搜索框中输入关键词，可输入项目名称、主题词、负责人、单位等信息。
3. 选择可选筛选条件，如来源、立项年份。
4. 提交查询请求。
5. 系统返回统一格式的项目列表。
6. 用户点击某条项目进入详情页。
7. 用户查看完整字段，并可点击原始来源链接跳转官方页面。

### 6.2 系统流程
1. 抓取脚本从指定官方来源拉取公开数据。
2. 系统解析原始页面、PDF 或接口返回内容。
3. 将各来源字段映射到统一标准字段。
4. 执行去重与更新逻辑，保存到数据库。
5. 若本地没有数据库连接，则将标准化结果写入本地快照文件。
6. 前端调用统一查询接口进行搜索。
7. 后端优先从数据库读取，否则回退读取本地快照并返回结果。
8. 前端渲染列表页与详情页。

## 8. 推荐技术栈
第一版建议采用单体 Web 应用方案，尽量减少工程拆分和部署复杂度。

### 8.1 应用层
- 前端框架：`Next.js`
- 语言：`TypeScript`
- UI：原生 CSS 或轻量组件方案
- 服务端接口：`Next.js Route Handlers`

### 8.2 数据层
- 数据库：`PostgreSQL`
- ORM：`Prisma`
- 无数据库兜底：本地 JSON 快照文件

### 8.3 抓取层
- 静态页面解析：`Cheerio`
- PDF 解析：`pdf-parse`
- 动态页面抓取：`Playwright`，预留给后续更复杂来源

### 8.4 调度与运行
- 第一版更新方式：手动执行抓取脚本
- 后续定时调度预留：`node-cron`
- 进程管理：`PM2`
- 反向代理：`Nginx`

## 10. 数据结构
### 10.1 统一项目字段

| 字段名 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| id | string | 是 | 系统内部主键 |
| source | string | 是 | 数据来源标识 |
| sourceProjectId | string | 否 | 来源侧项目唯一标识 |
| title | string | 是 | 项目名称 |
| principalInvestigator | string | 否 | 负责人 |
| institution | string | 否 | 所在单位 |
| discipline | string | 否 | 学科分类 |
| fundType | string | 否 | 基金类别 |
| year | number | 否 | 立项年份 |
| projectNumber | string | 否 | 项目编号 |
| status | string | 否 | 项目状态 |
| keywords | string | 否 | 关键词或主题词 |
| summary | string | 否 | 摘要 |
| sourceUrl | string | 否 | 原始详情链接 |
| sourceRaw | json | 否 | 原始抓取字段快照 |
| createdAt | datetime | 是 | 入库时间 |
| updatedAt | datetime | 是 | 最近更新时间 |

### 10.2 抓取记录表

| 字段名 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| id | string | 是 | 主键 |
| source | string | 是 | 数据来源 |
| startedAt | datetime | 是 | 开始时间 |
| endedAt | datetime | 否 | 结束时间 |
| status | string | 是 | `success` 或 `failed` |
| totalFetched | number | 否 | 抓取总数 |
| totalSaved | number | 否 | 入库总数 |
| errorMessage | string | 否 | 错误摘要 |

### 10.3 去重规则
- 优先规则：`source + projectNumber`
- 回退规则：`source + title + principalInvestigator + year`

## 11. 接口设计
### 11.1 查询项目列表
`GET /api/projects`

参数：`q`、`source`、`year`、`page`、`pageSize`

### 11.2 查询项目详情
`GET /api/projects/:id`

### 11.3 抓取执行
通过脚本执行：

```bash
npm run crawl:moe
npm run crawl:nsfc
```

## 12. 错误处理
- 前端：查询失败、空结果、详情不存在、参数非法时展示清晰提示。
- 后端：统一返回 `400`、`404`、`500` 错误。
- 抓取层：单条附件或单条记录失败时记录错误并继续处理后续内容。
- 数据层：数据库未配置时自动回退到本地快照，不中断查询演示链路。

- 可以按来源筛选。

- 可以进入详情页查看完整字段。

- 无数据库时也能展示最近一次真实抓取结果。
