# Step 11 - 成果线索聚合与匹配评分

## 目标
为成果查询模块接入第一批“公开成果线索”，实现论文、著作、结项成果的基础展示，并给出关联项目的匹配依据或可信度。

## 输入
- Step 10 已完成的 `/api/outcomes` 接口与项目定位能力
- [design.md](C:\Users\admin\Documents\yangjingjing\design.md) 中的数据来源分级与匹配策略
- 可稳定访问的公开成果网页或本地样例成果数据

## 输出
- 成果线索标准结构
- 至少一种可运行的成果来源接入方式
- 成果结果列表页面
- 每条成果显示成果类型、来源、链接和可信度信息

## 不做内容
- 不抓取受限全文平台正文
- 不处理登录态、验证码和馆藏授权
- 不实现复杂全文检索
- 不做批量多平台统一采集调度
- 不实现评论、收藏、导出

## 实现要求
- 至少接入 1 类成果线索来源，来源可以是：
  - 高校官网成果页
  - 科研处公告页
  - 机构知识库公开页
  - 本地标准化样例成果数据
- 成果线索字段至少包含：
  - `title`
  - `authors`
  - `outcomeType`
  - `publishYear`
  - `sourcePlatform`
  - `sourceUrl`
  - `matchedProjectTitle`
  - `matchedProjectNumber`
  - `matchedBy`
  - `confidenceScore`
- 匹配逻辑至少实现两层：
  - 批准号精确匹配
  - 项目名称匹配
- 页面需支持成果类型筛选：
  - 论文
  - 著作
  - 结项成果
- 前端需对可信度进行可读展示，例如：
  - 高可信
  - 中可信
  - 低可信
- 若接入的是样例成果数据，也必须是结构真实、字段完整、可解释的数据，而不是无意义占位符

## 校验方式
- 查询一个已知项目时，可以返回至少一批成果线索
- 成果列表中能看到标题、类型、来源平台和原始链接
- 至少一条成果具备明确的匹配方式说明
- 切换成果类型筛选后，结果可变化或正确为空
- 执行 `npm run build` 或 `npm.cmd run build` 成功

## 完成标准
- 成果查询模块首次具备“真实可浏览的成果线索结果”
- 用户可以根据成果类型、来源和可信度初步判断成果相关性
- 后续 Step 12 只需补外部平台跳转和整体收口，不需要重写成果结构
