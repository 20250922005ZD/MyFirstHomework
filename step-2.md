# Step 2 - 数据库模型与数据约束

## 目标
建立统一的数据模型和约束规则，确保后续抓取入库、查询接口和去重逻辑都基于同一套结构实现。

## 输入
- [design.md](C:\Users\admin\Documents\yangjingjing\design.md) 中的数据结构定义
- Step 1 产出的可运行工程骨架
- 可连接的 PostgreSQL 数据库
- 已配置的数据库连接环境变量

## 输出
- `Prisma schema`
- `projects` 表模型
- `crawl_runs` 表模型
- 必要索引和唯一性约束
- 可执行的数据库迁移文件
- 最小化的数据库连接验证结果

## 不做内容
- 不实现真实抓取
- 不实现查询页面
- 不实现 API 业务逻辑
- 不新增统计分析类数据表

## 实现要求
- `projects` 表至少包含设计文档列出的统一字段。
- `crawl_runs` 表至少包含抓取执行记录相关字段。
- 为高频查询字段建立索引，至少包括：
  - `source`
  - `year`
  - `title`
  - `principalInvestigator`
  - `institution`
- 落实去重约束策略：
  - 主规则使用 `source + projectNumber`
  - 回退规则使用 `source + title + principalInvestigator + year`
- 对允许为空的字段保持宽松，避免因来源字段缺失导致入库失败。
- 所有模型命名、字段命名和类型含义应与 `design.md` 保持一致。

## 校验方式
- `Prisma schema` 校验通过。
- 迁移命令可执行成功。
- 数据库中实际生成 `projects` 和 `crawl_runs` 两张表。
- 相关索引和唯一性约束可在数据库中核对到。
- 插入一条最小测试数据时，不会因为非必填字段缺失而失败。
- 数据库连接失败时能明确定位为环境变量或连接配置问题，而不是模型定义问题。

## 完成标准
- 后续步骤可以直接使用数据库模型，不再回头改表。
- 去重与更新规则已有明确承载位置。
- 抓取模块和 API 模块对数据结构的理解保持一致。
- 本步骤产物足以支撑后续入库和查询开发，无需再补数据库层前置工作。
